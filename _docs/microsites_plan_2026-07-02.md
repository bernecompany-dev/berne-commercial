# Сеть гео-микросайтов — план структуры (2026-07-02)

Статус: черновик на согласование. Ветка `claude/microsite-structure-options`.

## Цель

2–3 самостоятельных сайта в нише ремонта коммерческой техники, которые:

- медленно набирают индекс и ловят **дополнительный** органик-трафик (не каннибализируя Berne);
- являются **продаваемым активом** — по отдельности или в пакете с компанией.

## ⚠️ Главное правило (иначе всё зря)

**НЕ клонировать существующие сайты с заменой текста.** Google штрафует дубль-контент
и сетки доменов с общим футпринтом (дорвеи) — такие сайты не индексируются или банятся.

Каждый сайт = **уникальный контент + свой бренд/логотип/тон + разный технический
футпринт**. Только тогда он реально ранжируется и продаётся. Клон = потраченные деньги.

Из этого правила следует три требования к каждому сайту:

1. **Свой бренд** — имя, логотип, палитра, тон. Это **НЕ Berne** и не отсылка к Berne.
2. **Уникальный контент** — не рерайт наших страниц, а свой угол, свои услуги-акценты,
   свои гайды, свой FAQ, свои локальные ориентиры.
3. **Разный тех-футпринт** — другой генератор, хостинг, аналитика, формы, регистратор,
   вёрстка. Одинаковый стек на трёх доменах = очевидная сетка (см. матрицу ниже).

---

## Почему «954» — нет, а «по округам» — да

Идея бренда вокруг кода **954** отклонена:

- 954 — это код **только Броварда**. Бренд вокруг него отрезает Miami-Dade (305/786)
  и Palm Beach (561) — 2 из 3 округов зоны обслуживания.
- Не масштабируется: один код = один регион, дальше расти некуда.
- Это брендовый номер, а не структура сайта — SEO-охвата не даёт.

Вместо этого — **три гео-микробренда по округам**, каждый гиперлокален там, где основные
сайты слабее. При продаже уходят по отдельности или в пакете.

| Округ | Коды | Ядро рынка | Конкуренция |
|-------|------|-----------|-------------|
| Palm Beach | 561 | West Palm, Boca, Jupiter — отели, кантри-клубы, курорты | ниже всего |
| Broward | 954 | Fort Lauderdale — рестораны, склады, марины | средняя |
| Miami-Dade | 305/786 | Brickell, Wynwood, Doral, Hialeah — крупнейший, двуязычный | выше всего |

## Три бренда (концепты)

Имена — **кандидаты на выбор**, финал за тобой. Все три — независимые, локальные,
с разным тоном, чтобы не читались как один автор.

### 1. Palm Beach (561) — премиум/гостеприимство

- **Тон:** white-glove, спокойный, «сервис для отелей и клубов».
- **Акцент услуг:** рефрижерация, льдогенераторы, кухни ресторанов при отелях, сервис-контракты.
- **Кандидаты имени:** `Intracoastal Commercial Repair` · `Palm Coast Kitchen Service` · `Loxahatchee Refrigeration Co.`

### 2. Broward (954) — быстрый/надёжный

- **Тон:** без воды, «приедем сегодня», рабочий, по делу.
- **Акцент услуг:** аварийный ремонт 24/7, рестораны, склады, марина-оборудование.
- **Кандидаты имени:** `New River Refrigeration` · `Sawgrass Kitchen Service` · `Everglade Commercial Repair`

### 3. Miami-Dade (305/786) — двуязычный/B2B-рестораны

- **Тон:** энергичный, EN + ES наравне (большой испаноязычный рынок), city-vibe.
- **Акцент услуг:** коммерческие кухни, эспрессо/кофе, льдогенераторы, fryer/hood — плотный HoReCa.
- **Кандидаты имени:** `Biscayne Commercial Kitchen Repair` · `Magic City Coldline` · `Vela Refrigeration`

## Матрица разного тех-футпринта (критично)

Чтобы три сайта **не** выглядели как одна сетка, они должны различаться по максимуму
технических сигналов. Никаких общих: хостинг-аккаунта, GA-аккаунта, шаблона, шрифта,
регистратора, IP-диапазона, robots/sitemap-паттерна.

| Сигнал | Site 1 — Palm Beach | Site 2 — Broward | Site 3 — Miami-Dade |
|--------|---------------------|------------------|---------------------|
| Генератор | Astro (static) | Eleventy (11ty) | Hugo |
| CSS | Tailwind | vanilla CSS / PicoCSS | Bulma |
| Хостинг | Cloudflare Pages | Netlify | Vercel |
| Формы | Web3Forms | Netlify Forms | Formspree |
| Аналитика | Plausible | Fathom | GA4 (отд. аккаунт) |
| Регистратор | отд. #1 (privacy WHOIS) | отд. #2 | отд. #3 |
| Шрифты | Source Serif + Inter | System UI stack | Poppins + Lora |
| IA / навигация | hub-and-spoke по услугам | плоская + блог-heavy | город→услуга сетка |
| Логотип/палитра | сине-песочный, серифный | зелёно-графитовый, гротеск | бирюза+магента, geometric |

> Это НЕ значит «три разных стека ради красоты» — это значит **разные отпечатки**, чтобы
> кросс-доменная связь не читалась. Контент, тексты, фото — тоже свои под каждый бренд.

### Что должно быть РАЗНЫМ в контенте (не клон)

- Разное наименование и группировка услуг (не тот же список слаг-в-слаг).
- Разный формат лонгридов: PB → сервис-контракты/maintenance; BR → emergency/24-7; MD → buying-guides + двуязычие.
- Свой FAQ под каждый бренд (разные вопросы, не рерайт).
- Разные локальные ориентиры/районы, свои телефоны и адрес (registered agent / virtual office по округу).
- Свои фото/иллюстрации, своя тон-оф-войс.

## Порядок запуска

Начинаем с **наименее конкурентного округа** — быстрее первый индекс-вин и обкатка процесса.

1. **Site 1 — Palm Beach (561)** — пилот. Ставим процесс: 8–12 уникальных страниц, свой бренд, свой хостинг.
2. **Site 2 — Broward (954)** — через 3–4 недели после индексации пилота.
3. **Site 3 — Miami-Dade (305/786)** — последним, самый объёмный и двуязычный.

Разнос по времени + разные регистраторы/хостинги = меньше следов сетки.

## Ограничение окружения (важно для реализации)

Разный футпринт требует **отдельных репозиториев, хостинг-аккаунтов и регистраторов** —
это операционные вещи вне этого репозитория (мой доступ ограничен `berne-commercial`).

Поэтому предлагаю так:

- **Здесь (в этой ветке):** этот план + готовый **стартер-скелет Site 1** как отдельный
  самостоятельный проект (папка вне `apps/`), который потом переносится в свой репозиторий и хостинг.
- **Ты/оператор:** заводишь отдельный репо + Cloudflare Pages + домен + регистратор под Site 1.
